Executive Summary

The analyzed code contains a High-Severity SQL Injection (SQLi) vulnerability. The application accepts user input directly into a database query string without sanitization. This flaw allows attackers to manipulate the query logic, potentially bypassing authentication, stealing sensitive user data, or destroying the database.

Vulnerable Code (Line 7) 


import sqlite3

db = sqlite3.connect("users.db")
cursor = db.cursor()

username = input("Enter username: ")

# ❌ VULNERABLE: Direct f-string injection
query = f"SELECT * FROM users WHERE username = '{username}'"
cursor.execute(query)

result = cursor.fetchall()
print(result)

Why is this dangerous?

Attack Vector

If an attacker enters: ' OR '1'='1

Resulting Query

SELECT * FROM users WHERE username = '' OR '1'='1'

Since '1'='1' is always True, the database returns ALL users instead of just one.

CWE Classification

CWE-89: Improper Neutralization of Special Elements in SQL Command

Recommended Fix

secure_patch.py Parameterized Query Approach 

# ✅ SECURE: Use Parameterized Queries (Placeholders)
# The database driver handles the escaping automatically.

# Instead of f-string:
query = "SELECT * FROM users WHERE username = ?" 

# Pass variables as a tuple in the second argument:
cursor.execute(query, (username,))

Professor's Note: Performance & Security

Using parameterized queries not only prevents SQL Injection but allows the database engine to compile and cache the query plan (Execution Plan).

If you execute the same query 1,000 times with different usernames:

  • f-strings: Database sees 1,000 different queries. It must re-compile the plan 1,000 times. (Slower)
  • Parameters: Database sees 1 query structure. It compiles once and reuses the plan. (Faster)
